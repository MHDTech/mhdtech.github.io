TECHNICAL #1: Thoughts on Polymer

Hi Everyone again! It's been just few hours after first blog post i have made earlier. So let's get for technical for this one. Not much coding, just preview and no worries.

For those who have known Google released interesting design guideline called Material Design (MD), which aimed to unified all frameworks across any devices including desktop, smartphones. So what i have been doing in order to implement MD. I installed Android Studio 1.4 for mobile apps development, tried Polymer for web app development using their starter kit. Building Android apps is simple when i got used to it, Everything is awesome ! Just one thing bout the web components libraries i really getting excited to test it out: Polymer.

Until one day..

So i tried to use Polymer Starter Kit (PSK) and Material Design Lite for trying to building a site instead of mobile web app, porting components, writing contents. Until i came down to fews problem that conclude: the layout of the page itself. Polymer components are built to be accomodated with our page called :host. Therefore, i dealt alot things regarding starter kit such as:

  1. Padding/Margins: For example, i ported an element called paper-card (aka card with just content) like below:
    2. <html>

                            <head>
                              <base href="http://polygit.org/polymer+:master/components/">
                              <link href="http://polygit.org/polymer+:master/components/paper-material/paper-material.html" rel="import">
                            </head>

                            <body class="fullbleed">
                              <paper-material elevation="1">
                                <div>Hello SC!</div>
                              </paper-material>
                            </body>

                            </html>

    See the Pen Paper-Material Polygit by SC Kim (@SCKim) on CodePen.

    Everything supposedly worked fine, but what if the predefined styles gone wrong, yet we can't just edit using styles nor hardcoded. What would be the worst is to touch the library's code itself? I wouldn't wanna do that every time.

  2. Cross-Platform: We have two ways to import components. Either we provides the components ourselves via bower to host, or we can directly callback import into our html from Polygit (Server of Polymer Components) which hosted in Github. Polygit still great yet light for us to import but make us dependable. Because i do concern about page speed and performance so i choose 1st method. Good news is most of the components can work in multiple browsers using Polyfills for custom Elements, HTML Imports and Shadow DOM ! Which is better? I can't decide for you, is the best for you to try it out

  3. Components Dependencies: Material Design components do have multiple layers of components dependent to each other. For example, a FAB (Floating Action Button) has at least three layers of components reused: <paper-ripple> , <iron-icon>, <paper-material>. So if you added some styling or making changes one of the components. You will get something like this, ouch? That's not it, assuming every components that you used will be affected along the changes: DISASTROUS.

  4. Panel structures, it's abit messed up: If you wished to create site with app-drawer/navigation drawer, Polymer did provided few: <paper-header-panel> , <paper-scroll-header-panel>, <paper-drawer-panel> When you used any of the two combined, you will get most of the layout well-designed. However, you will pissed off when u scroll your contents like this in any mobile browsers.

    How did it happened? Why Chrome Address Bar doesn't hide after scroll! Even Safari and Firefox mobile browser showing same anomalities. No matter what kinda magic hacks of Javascript, styling, meta-tags or any adjustments, you still can't get it UP!

    5. Buggy Address Bar
    So here's how that panel components work for my hypothesis. Every content of yours will be contained under panel itself instead of being independent. To be more evident, you may view the screnshoots below. The blue themed was build using the panel elements while red-themed was build without the panel, you may see significant differences in left-rulerview or dark-overflow contents below the red-themed page.

    That's why browser doesn't recognize the contents is whether overflowed or just fit the screen. So the whole layout will never be overflow and the address bar will never dissapear no matter how hard you've scrolled. So what's the solution then? Don't use the panels, refactor yourself. I took few days to figure it out until i check other overflow mobile sites with F12 Chrome Developer Tools..Dang..Although you may seek this as small matters in phablet/giant size phones, but as a web developer, it is essential to assume our mobile sites are accessible yet convenient even for small sizes like 3.5-inches size phones.
    Overflow Failed (Blue) and Overflow (Red)

  5. Routing : Depends how you starts with your application, if you starts from Polymer Starter Kit, which will focus on single page app, instead of loading multiple-pages. It's similar as Fragment in Android Activity. So if your href is mismatch as routing.js or you "open link in new tab" in PSK. You'll most probably get error.

  6. MD Guidelines : Not all components follow completely guidelines & examples of Material Design. Some might missed typography, measurements or components positioning and etc.

So, what's the conclusion and advice for future Polymer starters?

So what'd i like to give an advice if you planned to use the components:

  • First, check the components whether following the guidelines, otherwise send a issue to github so they could review it.
  • Next, you need your F12 key alot to debug the components and check layer-by-layer embedded within it.
  • Lastly if you don't feel like to use the components, make your new one! That's why Polymer is awesome !
